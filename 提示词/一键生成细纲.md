## 使用方法与自动探测机制
1. **指定指令**: 明确指定起始章节号（例：`@[/一键生成细纲] 41-45章`）。
2. **自动探测 (无参触发)**: 若用户仅输入 `@[/一键生成细纲]` 未做任何指令补充或说明，AI 必须强制执行以下动作：
   - 自动扫描项目内的 `06_大纲细纲库` 文件夹及所有子卷目录，识别当前最高/最新的细纲记录（例如：第45章细纲）。
   - 根据自动找出的最新记录，自动向后顺延，设定紧接着的 **连续5章**（例如46-50章）为本次生成目标。
3. AI 再自动读取所有必备上下文，并按节奏规范生成这连续5章的细纲。
4. 自动保存到对应卷的细纲文件夹。

## 执行步骤

### 步骤1: 强制清空记忆假定与物理调用（防漏机制）

> [!CAUTION]
> **【多模型切换防崩坏机制 - 头等执行优先】**
> 你必须进行“零记忆体系假定”：无论你觉得你是否在之前的对话中见过这些文件，都**假定你现在处于丢失上下文的全新状态**。因为你极有可能是刚刚被主系统唤醒调度来承接当前步骤的全新模型实例（与其他模型间不共享记忆缓存）。
> **绝对禁止**依赖内置知识或虚假幻觉进行瞎编！
> 你**必须且只能**通过具体的工具调用（Tool Calls）去系统文件目录下**真实物理读取**以下表格中的所有必读文件，这是强制工具调用链。

在进行任何思考和输出前，排队调用工具读取以下所有 `⭐⭐⭐` 必读清单：

**【强制阻断点】**：在完成所有的 API 读取操作后，你必须首先在对话框内输出一个**「📁 物理读取完成清单」**打勾向用户汇报，并在用户默认通过后（或无障碍自动连贯时）才允许进入下方的 `步骤2` 环节。

### 步骤2: 确定细纲范围
- 根据指定起始章节号生成5章

- 确定核心剧情走向

### 步骤2.5: ⚡节奏锚定与进度控制（强制执行）⚡

> [!CAUTION]
> **这是本工作流最重要的约束步骤，必须在生成细纲之前完成计算，并严格遵守结果。**
> 违反此步骤的细纲将被视为无效，必须重新生成。

#### 2.5.1 读取整书大纲，计算进度比例

打开 `00_核心设定/整书大纲.md`，找到当前5章所属的卷，记录以下信息：

| 字段 | 示例 |
|------|------|
| 当前卷名 | 第一卷：XXX |
| 本卷章节范围 | 第1章 ~ 第X章 |
| 本卷总章节数 | X章 |
| 本次细纲起始章 | 第X章 |
| 当前进度百分比 | X/总章数 = **X%** |
| 本次5章结束后进度 | (X+5)/总章数 = **X%** |
| 本次5章推进幅度 | **X%** |

#### 2.5.2 锚定本卷剧情推进量

根据`整书大纲.md`中本卷的「主要剧情」列表，将本卷所有核心事件**均匀分配**到整卷章节中：

**计算公式**：每个核心事件应占 = 本卷总章节数 ÷ 核心事件数

例如某卷有5个核心事件、150章：
- 每个核心事件应占约 **30章**
- 5章细纲最多只能推进 **1/6个核心事件** 的进度

> [!WARNING]
> **绝对禁止**：5章细纲内完成一个完整的核心事件！
> 核心事件至少需要20-30章来展开，每5章只应推进该事件的某个阶段。

#### 2.5.3 强制填充机制

每5章细纲中，**必须包含至少2章**属于以下类型（不能全是主线高潮）：

| 填充类型 | 说明 | 示例 |
|----------|------|------|
| 🔀 **支线展开** | 独立于主线的次要剧情，丰富世界观 | 配角的个人故事、社会环境变化 |
| 🧠 **人物刻画** | 通过日常/互动展现人物性格深度 | 主角与身边人的相处、内心独白 |
| 🎭 **伏笔铺垫** | 为未来10-20章的事件做预先布局 | 提前接触关键人物、偶然目睹某事 |
| 🌊 **氛围过渡** | 时代背景、社会环境的细腻描写 | 历史大事件对故事世界的影响 |
| ⚔️ **小冲突** | 独立的小型冲突和解决，不涉及核心事件 | 日常矛盾、小范围对抗 |

#### 2.5.4 节奏分级规则

根据当前进度百分比，适用不同的节奏规则：

| 进度区间 | 节奏要求 | 主线占比上限 |
|----------|----------|--------------|
| 0%-20%（卷初） | 慢热铺垫，大量日常与人物建立 | ≤40%（5章中最多2章推主线） |
| 20%-60%（卷中） | 稳步推进，支线穿插 | ≤60%（5章中最多3章推主线） |
| 60%-80%（卷末前奏） | 逐步加速，收束支线 | ≤80%（5章中最多4章推主线） |
| 80%-100%（卷末高潮） | 全面加速，高潮迭起 | 100%（可全部推主线） |

#### 2.5.5 输出进度锚定表

在开始生成细纲前，必须先输出以下锚定表供确认：

```
=== 节奏锚定确认 ===
当前卷：[卷名]
本卷总章数：[X]章
当前起始章：第[X]章
当前进度：[X]%
所处阶段：[卷初/卷中/卷末前奏/卷末高潮]
当前核心事件：[大纲中的第X个核心事件]
该事件已推进章数：[X]章
该事件预计总章数：[X]章
本次5章可推进主线章数：[X]章
本次5章填充章数：[X]章
==================
```

### 步骤3: 生成5章细纲（必须严格遵守步骤2.5的锚定结果）
每章细纲使用以下模板:

```markdown
# 第X章细纲

## 基本信息
- **章节名**: [吸睛标题，体现核心冲突或反转]
- **场景**: [具体地点]
- **核心人物**: [本章主要出场人物]

## 剧情概括
[一段话概括整章剧情，包含起因、经过、结果]

## 核心事件点
1. [事件1：具体描述]
2. [事件2：具体描述]
3. [事件3：具体描述]

## 爽点设计
- **类型**: [迪化/打脸/逆袭/躺赢/反差]
- **描述**: [具体爽点场景]
- **效果**: [读者预期反应]

## 涉及的伏笔
- 【埋伏笔】[伏笔名称]: [具体内容]
- 【收伏笔】[伏笔名称]: [如何回收]

## 章末钩子
- **类型**: [悬念/期待/转折/信息/冲突]
- **内容**: [具体钩子描述]

## 预估字数
- [2200-3000字]

## 本章节奏类型
- **类型**: [主线推进/支线展开/人物刻画/伏笔铺垫/氛围过渡/小冲突]
- **理由**: [为什么本章选择此节奏类型]

## 与前后章衔接
- **承接上章**: [如何接续上一章结尾]
- **引向下章**: [为下一章做什么铺垫]
```

### 步骤4: 节奏检查
确保5章整体满足:
- 每章都有独立爽点
- 整体有递进高潮
- 第5章有较强钩子（小高潮）
- 伏笔埋设和回收平衡

**⚡ 进度合规检查（必须全部通过）：**
- [ ] 主线章数是否符合步骤2.5.4的节奏分级规则？
- [ ] 填充章数是否 ≥ 2章？（卷末高潮阶段除外）
- [ ] 是否有核心事件在5章内被完整完成？（若是则必须拆分）
- [ ] 5章总共推进的剧情量是否与进度比例匹配？
- [ ] 是否有新的支线或伏笔被引入？（每5章至少引入1条新支线或伏笔）

> 如果任何一项不通过，必须回到步骤3重新调整细纲。

### 步骤4.5: ⚡强制逻辑审查与修复（绝对阻断点）⚡

> [!CAUTION]
> **【生成后强制拦截机制】**
> 本步骤为核心阻断点，不准你出于“AI直觉”自认为没有问题而蒙混过关！
> 在审核前，**你必须第一步先触发工具真实物理读取：**
> `00_核心设定/审核规则.md` 
> 去获取项目中独家的审查底线红线记录表。没有在后台调取这个文件就直接打印报告，视为作弊与拒不执行。

#### 4.5.1 标准审核清单

对刚生成的5章细纲，逐一检查以下7大类问题：

| # | 检查维度 | 检查内容 | 常见问题 |
|---|----------|----------|----------|
| 1 | **时间线连贯** | 核对`时间线发展记录.md`，章节所处年份是否与时间线一致？人物年龄是否正确？ | 年龄写错、年份跳跃 |
| 2 | **时代物品检查** | 正文中出现的物品、科技、交通工具是否符合当前年代？ | 年代不符的物品出现 |
| 3 | **职级/程序合规** | 官职升迁、罢免、任命等是否符合体制程序？ | 直接罢免缺程序包装 |
| 4 | **人物能力边界** | 各角色的行动是否在其身份/资源能力范围内？ | 小角色拥有越界能力 |
| 5 | **章间衔接检查** | 每章开头是否与上一章结尾衔接？时间/地点/人物状态是否连贯？ | 前章推进一个月，后章回退到刚开始 |
| 6 | **称谓/数据一致性** | 同一人物/事件的称谓、数字在不同章节是否一致？ | 同一事件描述不一致 |
| 7 | **伏笔逻辑检查** | 收回的伏笔是否在前文确实埋过？新埋的伏笔是否有回收计划？ | 凭空收回从未铺垫的伏笔 |

#### 4.5.2 自动修复流程

1. 按上述7大类逐章扫描，列出所有问题
2. 对每个问题标注严重度（🔴严重/🟡中度/🟢轻度）
3. 对🔴和🟡问题，**直接在细纲文件中修复**，无需等待用户确认
4. 对🟢问题，标注但不做修改（留给正文阶段处理）
5. 修复完成后，输出简要的审核报告（问题列表+修复措施）

#### 4.5.3 输出审核结果

```
=== 细纲逻辑审核报告 ===
审核范围：第[X]-[Y]章细纲
发现问题：🔴[X]个 🟡[X]个 🟢[X]个
已自动修复：[X]个
待正文阶段处理：[X]个

[问题1] 🔴 [章节] [问题描述] → 已修复：[修复措施]
[问题2] 🟡 [章节] [问题描述] → 已修复：[修复措施]
[问题3] 🟢 [章节] [问题描述] → 待处理
========================
```

### 步骤5: 保存细纲

## 番茄爽文节奏参考

### 5章整体结构
- **第1章**: 开局冲突，引入新问题
- **第2章**: 问题升级，压力增大
- **第3章**: 主角被迫行动，意外展开
- **第4章**: 局势反转，爽点高潮
- **第5章**: 阶段性解决+新钩子

### 每章内部节奏
- 开头: 承接上章，快速进入场景
- 前1/3: 铺垫冲突
- 中1/3: 冲突爆发，主角行动
- 后1/3: 结果揭晓，爽点释放
- 结尾: 留钩子

## 注意事项

1. 细纲要足够详细，能直接指导正文创作
2. 核心事件点要具体到对话和场景
3. 爽点设计要明确，不能模糊
4. 伏笔要标注清楚是埋还是收
5. 章节名要吸睛但不剧透
6. **【重要】每章细纲开头必须包含"核心设定提醒"区块**
7. **【强制】必须先完成步骤2.5的节奏锚定计算，再生成细纲**
8. **【强制】不可在5章内完成整书大纲中的任何一个完整核心事件**
9. **【强制】每5章必须至少有2章是支线/日常/伏笔/过渡章，避免全程高潮**
10. **【强制】每次生成细纲前必须输出「节奏锚定确认表」**
11. **【提醒】整书大纲中每个核心事件都应花20-30章展开，不要赶进度**
12. **【提醒】多创造新支线、新人物互动、新伏笔来充实剧情，不要只消耗大纲中已有的事件**
